# PochiClock - CLAUDE.md

## 概要
Smiley全部門向け勤怠管理Webアプリ。スマホ・PC操作が苦手な人でも使えるシンプルさが最優先。

**技術スタック**: Laravel 12 + Tailwind CSS + MySQL + UUID

## 要件（確定済み）

### 打刻方式（2方式併用）

| 方式 | 対象部門 | フロー | 工数 |
|------|---------|--------|------|
| PIN打刻 | 配食（調理・配達） | キオスクで名前タップ → 4桁PIN | 2ステップ |
| QR打刻 | 美容 | スマホQR → タブレットカメラ | 2ステップ |
| 未確定 | 介護・居宅・タクシー | 要検討（下記参照） | - |

- 配食部はスマホ苦手な人が多い → キオスク端末のみで完結するPIN方式
- 美容部は既にQR運用実績あり＋タブレットあり → QR方式継続
- **タブレットは美容部にしかない** → 介護・居宅・タクシーのQR打刻はカメラ付き端末の調達が前提。PCのWebカメラで代替するか、PIN方式にするか要検討
- ICカードは廃止（発行の手間＋紛失リスク）
- PIN忘れ → 管理者がリセットできる機能を実装

### キオスク設定
- 各キオスク端末にデフォルト部署を設定可能
- URL: `/kiosk/{department}` （現状の実装をそのまま活用）
- タブレットのホーム画面に部署URLをブックマーク登録するだけ
- 他部署の人が使う場合は部署切替ボタンで変更可能

### 部門別勤務ルール

#### 配食部門 - 調理
- 所定労働時間: 6h
- 1人早番、他は出勤時間同じ・退勤はまちまち
- **早出カット**: 8:00以前の打刻は8:00に丸める（自動適用）
- 個人別ルール例: 本城さん（早出+30分加算）

#### 配食部門 - 配達
- **2回出勤パターン**: 午前便（9:30〜）+ 午後便（14:00〜）
- 1日に出勤・退勤が2セットある → **打刻モデルの改修が必要**（現状1日1レコード想定）
- **早出カット**: 午前便9:30以前、午後便14:00以前を自動丸め
- 矢野さん: 12:30以降カット、筒井さん: 12:00以降カット

#### 介護部門 - 訪問介護
- **常勤のみPochiClock対象**（タイムカード「まいどシステム」から移行）
- 登録ヘルパーはカイポケのみで管理（スコープ外）
- 直行直帰が多い → **スマホPWA+GPS打刻**が最適
- 打刻漏れ → 未打刻アラート（翌日朝に一覧表示）

#### 介護部門 - 居宅介護支援
- 固定時間: 8:30-17:30
- 基本事務所勤務、訪問日だけスマホ打刻
- **残業→振替休日の可視化**が必要（個人の裁量に依存している現状を改善）
- 管理画面で「残業○h → 振替消化○h → 残り○h」を表示

#### 介護部門 - タクシー
- 居宅と同じ扱い（固定8:30-17:30、残業→振替の可視化）

#### 美容部門（Can I Dressy 寝屋川・守口）
- シフト制: 早番・中番・遅番・アルバイト
- **出勤15分前**までは給与に含める
- **残業は事前申告制** → 申告なし＋シフト超過 → アラート表示
- サロンボードとの退勤時刻突き合わせ → **API非公開のため自動化不可**。シフト超過の自動検出で代替
- 寝屋川・守口で勤務形態は同じ

### 勤務ルール設計
- 「部門」ではなく**「職種グループ」単位**で設定
- 例: 「配食-調理」「配食-配達」「訪問介護」「居宅」「美容」など
- 現状のWorkRuleモデル（全社/部署/個人の3階層）を職種グループに読み替え

### 自動化する勤怠チェック（現在手作業）
- 打刻漏れ検出 → 未打刻アラート
- 早出カット → 勤務ルール設定で自動丸め
- 個人別ルール → 個人別WorkRuleで設定
- シフト超過検出 → アラート表示（美容部）
- 残業・振替の可視化（居宅・タクシー）

### TKC PX2連携
- CSV出力 → TKC PX2の「外部データ受入」機能でインポート
- 現状のCSV出力（9項目）にTKC向け項目（残業区分別、有給日数等）を追加する必要あり
- **要**: 萩原さん（顧問税理士）からPX2のレイアウト定義書を入手（タスク登録済み）

## 拠点情報

| 拠点 | 所在 | 対象部門 | 端末 |
|------|------|---------|------|
| スマイリー事務所1階 | メイン | 配食・タクシー・経理 | PC（タブレットなし） |
| スマイリー事務所2階 | 同建物 | 訪問介護・居宅 | PC（タブレットなし）+ スマホPWA |
| Can I Dressy 寝屋川店 | 遠隔 | 美容 | タブレット（QR打刻用） |
| Can I Dressy 守口店 | 遠隔 | 美容 | タブレット（QR打刻用） |

※ ショウはスマイリー事務所1階で勤務 → 不具合の検知・対応がしやすい

## 導入計画

### 導入順序（足元から固める）

| 順序 | 拠点・対象 | 理由 |
|------|-----------|------|
| 1st | スマイリー事務所1階（配食・タクシー） | ショウが目の前。不具合即対応。配食は人数多く効果大 |
| 2nd | スマイリー事務所2階（訪問介護・居宅） | 同じ建物。キオスク+スマホPWAの両方を試せる |
| 3rd | Can I Dressy 寝屋川・守口 | 安定してから遠隔拠点へ。タブレット対応も検証済み |

### 3段階検証フロー（給与直結のためミスは許されない）

| フェーズ | 内容 | 期間 | 判定基準 |
|---------|------|------|---------|
| 1. 過去データ検証 | まいどシステムの過去1-2ヶ月分を取り込み、TKC入力値と突き合わせ | 1-2日 | 計算結果が一致 |
| 2. 並行運用（ショウのみ） | まいど + PochiClock両方で打刻 | 1ヶ月 | 全営業日で一致 |
| 3. 並行運用（部門拡大） | 対象部門全員で並行打刻。まいどが正本 | 1ヶ月 | 全員分一致＋運用問題なし |

フェーズ3クリア後に初めてまいどシステムから切り替え。
